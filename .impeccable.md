# Bimonth — Design Context

This file captures the *non-visual* design context for Bimonth: who the
product is for, the implementation-level token-to-element mapping, and any
open follow-ups that haven't landed yet. The visual system itself (palette,
typography, spacing, components, do's and don'ts) is the source of truth in
[`DESIGN.md`](./DESIGN.md). The functional spec lives in
[`docs/spec.md`](./docs/spec.md).

## Users

macOS desktop users who want an always-visible, glanceable view of two
adjacent months — "the recent past and the upcoming future" — without
clicking forward or back. The widget lives on the desktop or in Notification
Center, read in peripheral vision in under a second. Users glance to orient
themselves in time, not to manage events.

**Glanceable over interactive.** The widget is read in under a second from
peripheral vision; every element must survive that constraint or be cut. The
container app stays minimal for the same reason — it exists to host the
widget extension, not to be a destination.

## Token-to-element mapping (resolved)

Brand tokens are defined in `Shared/Colors.swift` and shared between the
container app and the widget extension. The widget asserts identity through
a few branded elements; everything else stays on system semantic colors so
light/dark mode adapts without a custom dark palette.

| Element                                       | Token              |
|-----------------------------------------------|--------------------|
| Widget month title                            | `brandBark`        |
| Widget today circle                           | `brandSage`        |
| Widget today digit                            | `brandParchment`   |
| Widget body text (in-month weekday/header)    | `.primary` (system)|
| Widget body text (weekend, weekday header)    | `.secondary`       |
| Out-of-month cells                            | `Color.clear`      |
| Container app icon (`ContentView`)            | `brandBark`        |

`brandInk` and `brandStone` are defined for future use but not currently
mapped to any UI element — keep them in the palette so future surfaces have
the full vocabulary available.

## Open follow-up: dark-mode variants

`Shared/Colors.swift` currently holds only light-mode hex values. In dark
mode, `brandInk` and `brandBark` are too dark to read on a dark widget
background. The next step is to migrate the palette to Asset Catalog Color
Sets with explicit light/dark pairs (target hex values are listed in
[`DESIGN.md`](./DESIGN.md) under *Colors → Light & Dark*). `brandSage` and
`brandParchment` (the today highlight) read in both modes and do not need a
separate dark variant.
